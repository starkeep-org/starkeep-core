# Data Server

The data-server is the local hub for all Starkeep data operations. It embeds the SDK and exposes it over HTTP so that multiple apps on the same machine can share a single database, type registry, and sync connection.

Start the data-server after the admin panel has guided you through initial setup.

## Running

```bash
pnpm --filter @starkeep/data-server dev
```

Listens on port 9820 by default. Set `STARKEEP_PORT` to use a different port.

## What It Does

### Data Operations

Apps read and write records through the data-server's HTTP API. The data-server enforces access control on every operation and applies schema validation before writing to storage.

Registered record types and their record counts are available at `GET /data/types`.

### File Watching

The data-server can watch local directories and automatically index new files as records. When a file is added to a watched directory, the data-server creates a record for it and runs any applicable metadata generators (e.g., extracting EXIF data from a photo).

Watches are managed via the API:
- `GET /watches` тАФ list active watches
- `POST /watches` тАФ add a new watch (directory path, target record type, recursive flag)
- `DELETE /watches/{id}` тАФ remove a watch

Watch configuration is persisted to `~/.starkeep/watches.json` and restored on startup.

### Authentication

The data-server manages Cognito credentials for connecting to the user's cloud infrastructure. Tokens are stored at `~/.starkeep/auth.json` and refreshed automatically.

- `POST /auth/tokens` тАФ store a Cognito token
- `GET /auth/status` тАФ check authentication state
- `POST /auth/logout` тАФ clear stored credentials

### Configuration

Cloud connection details (Aurora DSQL endpoint, S3 bucket, API Gateway URL, Cognito pool IDs) are read from `starkeep-config.json`. This file is generated by the admin panel's cloud setup wizard.

The current configuration is readable at `GET /config` and updatable at `PATCH /config`.

### Sync

When cloud credentials and configuration are available, the data-server syncs automatically:
- **Pull** тАФ fetches remote changes on a configurable interval (default: every 30 seconds)
- **Push** тАФ sends local changes to the cloud shortly after they occur (default: 500ms debounce)

Sync runs in the background. Apps do not need to trigger it manually.

### File Storage

Files attached to records are served via the data-server's `/files` routes. Signed URLs are issued for direct access and are invalidated when the data-server restarts.

## Configuration

Runtime settings live in `~/.starkeep/config.json` (or `$STARKEEP_DIR/config.json`). The file is auto-created on first boot with a generated `nodeId`; cloud fields are filled in later by the admin panel's cloud setup wizard via `PATCH /config`.

Recognised keys: `nodeId` (generated once, never change), `pullIntervalMs` (default 30000), `pushDebounceMs` (default 500), `stage`, `userPoolId`, `userPoolClientId`, `identityPoolId`, `s3Bucket`, `s3Region`, `auroraEndpoint`, `apiGatewayUrl`.

Only two env vars are recognised тАФ both bootstrap concerns that can't live in the config file:

| Environment Variable | Default | Description |
|---|---|---|
| `STARKEEP_PORT` | `9820` | HTTP port |
| `STARKEEP_DIR` | `~/.starkeep` | Root directory for local data (chicken-and-egg with the config file's own location) |

## Local Storage

| Path | Contents |
|---|---|
| `~/.starkeep/data.db` | SQLite database (all records and metadata) |
| `~/.starkeep/objects/` | File attachments |
| `~/.starkeep/auth.json` | Cognito tokens |
| `~/.starkeep/watches.json` | File watch configuration |
| `~/.starkeep/cloud-credentials.json` | STS credentials for cloud access |
