# Sync Test Constraints

Compatibility predicates between dimensions from `sync-test-dimensions.md`.
Used to prune the Cartesian product before enumeration.

Notation:
- `P(x)` = predicate true for value `x`
- Each section lists *only* the constraints; any combination not excluded is allowed.
- "Side" = `cloud` or `local`.

---

## C1. Operation requires compatible initial presence

Operations are `{side}-{verb}` where verb ∈ {`insert`, `update`, `soft-delete`}; the side performing the action must hold (or not hold) the record accordingly. Applies uniformly to all three data types.

| Verb | Required presence on the acting side |
|---|---|
| `insert` | acting side does NOT have the record (presence ∈ `neither` or `<other-side>-only`) |
| `update` | acting side HAS the record (presence includes the acting side) |
| `soft-delete` | acting side HAS the record |
| `concurrent-update-both-sides` | both sides have the record (`both-same` or `both-diverged`) |

Rationale: you cannot update or delete a record that does not exist on the side
performing the action; you cannot insert one that already exists there.

## C2. Tombstone state requires compatible initial presence

| Tombstone state | Required initial presence |
|---|---|
| `not-deleted` | any |
| `cloud-deleted` | `cloud-only`, `both-same`, or `both-diverged` |
| `local-deleted` | `local-only`, `both-same`, or `both-diverged` |
| `both-deleted` | `both-same` or `both-diverged` |
| `conflict-deleted-vs-updated` | `both-same` or `both-diverged` |

Rationale: a `deletedAt` on a side requires the record to exist on that side.

## C3. Tombstone state interacts with operation

Forbidden pairs (the operation cannot meaningfully drive a state already in this tombstone configuration):

- `{side}-soft-delete` when that side is already deleted (`<side>-deleted` or `both-deleted`)
- `{side}-update` when that side is already deleted (tombstone resurrection — model explicitly via `conflict-deleted-vs-updated` instead)

`conflict-deleted-vs-updated` is generated by `concurrent-update-both-sides` where one side's "update" is actually a soft-delete; treat that as its own operation variant rather than a free combination.

## C4. Data type constrains blob state

| Data type | Allowed blob states |
|---|---|
| `shared-record` | `cloud-has-blob`, `local-has-blob`, `both-have-blob`, `blob-key-present-but-missing` (blob is mandatory — `no-blob` is excluded) |
| `app-record` | all five blob states allowed (apps may choose per record whether it has a blob; blob lives in the app-specific S3 namespace) |
| `app-row` | `no-blob` only |

Rationale: shared-records are file-backed by definition (blob mandatory in the
shared S3 namespace). App-records may be file-backed or metadata-only at the
app's discretion (blob, when present, lives in the app's S3 namespace).
App-rows are app-namespaced syncable metadata tables, DB-only.

## C5. Data type constrains operation

All three data types accept the full operation grammar (`{local|cloud}-{insert|update|soft-delete}` and `concurrent-update-both-sides`). The only data-type-specific restrictions are on blob state (C4) and on failure modes that require a blob (C7).

## C6. Blob state requires compatible initial presence

A blob can only exist on a side where the record exists.

| Blob state | Required initial presence |
|---|---|
| `no-blob` | any (and required for DT=`app-row`) |
| `cloud-has-blob` | `cloud-only`, `both-same`, or `both-diverged` |
| `local-has-blob` | `local-only`, `both-same`, or `both-diverged` |
| `both-have-blob` | `both-same` or `both-diverged` |
| `neither-has-blob` | any presence where the record exists on at least one side (i.e. not `neither`); represents both-Staged for blob-bearing records |

## C7. Failure mode requires compatible operation / blob state

| Failure mode | Required context |
|---|---|
| `no-failure` | any |
| `fail-before-request` | exchange count ≥ `one-round` |
| `fail-after-send-before-response` | exchange count ≥ `one-round`; operation produces *some* outbound payload (i.e. not `zero-rounds`) |
| `blob-upload-fails` | the side initiating exchange has a blob the counterparty lacks (`local-has-blob` for local→cloud push, `cloud-has-blob` for cloud→local push) AND data type carries a blob (`shared-record`, or `app-record` with a non-`no-blob` state) |
| `blob-upload-fails-middle` | requires batch shape `multi-homogeneous` or `multi-mixed-nodes`; same blob preconditions as above |
| `blob-download-fails` | counterparty has a blob this side lacks; data type carries a blob (`shared-record`, or `app-record` with a non-`no-blob` state) |
| `partial-response-truncated` | requires batch shape `exceeds-page-limit` *or* multi-record state where the server can legitimately set `hasMore` |

## C8. Exchange count interacts with failure mode and recovery

- `zero-rounds` forbids every failure mode except `no-failure` (nothing ran, nothing could fail) and forces recovery to N/A.
- `until-converged` + any non-`no-failure` mode requires recovery (Dimension 10) = `transient`. `persistent` + `until-converged` is excluded (would never terminate).
- `one-round` admits both `transient` and `persistent` recovery, but recovery only becomes observable on round ≥ 2.
- `two-rounds` + `partial-response-truncated` is the canonical pagination test; `one-round` + `partial-response-truncated` is valid but tests only "watermark advanced partway".

## C9. Watermark state interacts with initial presence

- `zero` watermarks are only coherent with initial presences that could arise from a fresh start: `neither`, `local-only`, `cloud-only`. `both-same` + `zero` is contradictory unless we model "same content, never exchanged" (rare — exclude unless explicitly seeded).
- `current` watermarks imply prior convergence; only coherent with `both-same`, `neither`, or post-soft-delete `both-deleted` with matching `deletedAt`.
- `partial` is the catch-all and is compatible with any presence.
- `cloud-reset` and `local-reset` are compatible with any presence — they specifically test the case where data exists but bookkeeping doesn't.

## C10. Batch shape interacts with record count assumptions

- `single` forbids `multi-mixed-nodes`, `blob-upload-fails-middle` (needs a middle), and `partial-response-truncated` (one record never paginates).
- `multi-homogeneous` requires ≥2 records sharing a `nodeId`.
- `multi-mixed-nodes` requires ≥2 distinct `nodeId`s across the batch.
- `exceeds-page-limit` requires >2000 records — generator should treat this as a parametric case (synthesize N records) rather than enumerating each.

## C11. Concurrent operation has its own shape

`concurrent-update-both-sides` is the only operation that mutates state on *both* sides between exchanges. It implicitly requires:

- initial presence ∈ {`both-same`, `both-diverged`}
- tombstone state ∈ {`not-deleted`, `conflict-deleted-vs-updated`} (the latter when one "update" is a delete)
- exchange count ≥ `one-round` (otherwise no LWW resolution can be observed)

## Resolved scoping decisions

Decisions made while building this constraint model:

1. **Blob-key mismatch between sides** — out of scope. Not modeled.
   - Related: per-record blob residency (Absent/Staged/Resident/Tombstoned per side) is fully captured by the existing blob-state values; see `residency.ts` and `system-design.md` "Per-record residency". The old `blob-key-present-but-missing` value was removed (it conflated out-of-band corruption with normal Staged state).
2. **Failure recovery (transient vs persistent)** — added as Dimension 10 in `sync-test-dimensions.md`; referenced in C8.
3. **Derived-child / parent-child data type** — dropped entirely. From the sync engine's point of view records are records; parent/child is application semantics. Dimension 5 now lists only `shared-record` and `app-syncable-row`.
4. **HLC clock skew** — out of scope. HLCs are assumed to Just Work.
5. **Cloud-originated app writes** — in scope. Dimension 7 now uses a `{side}-{verb}` grammar where `side ∈ {local, cloud}` for all three data types, since cloud-resident apps can originate writes to app-records and app-rows.
6. **Three-way data type split** — `app-syncable-row` was too narrow. Dimension 5 now distinguishes `shared-record` (blob mandatory, shared S3 namespace), `app-record` (blob optional, app S3 namespace), and `app-row` (DB-only metadata). C4 updated.
