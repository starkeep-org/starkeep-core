# Sync Test Candidates (Phase 2)

Candidate test cases generated by expanding the 7 cross-cutting seeds from
`sync-test-dimensions.md` along compatible secondary dimensions, with
incoherent cells pruned per `sync-test-constraints.md` (C1–C11).

Each row is a candidate. IDs are stable (`<seed>-<n>`) so Phase 3 ranking can
attach scores by ID without renumbering. Phase 3 will keep ≥100 after dedup
and ranking.

**Companion docs (read first if approaching this fresh):**
- `sync-test-dimensions.md` — the 10 dimensions of variation.
- `sync-test-constraints.md` — pairwise compatibility predicates C1–C11.
- `../../../system-design.md` § "Per-record residency" — the Absent/Staged/Resident/Tombstoned model that Dimension 3 (blob state) encodes.
- `../src/residency.ts` — the `residencyOf` helper tests should assert on, not re-derive.

Conventions:
- DT = data type: `SR` (shared-record), `AR` (app-record), `AW` (app-row).
- Side = `L` (local) | `C` (cloud).
- Tomb state abbreviations: `nd` (not-deleted), `cd` (cloud-deleted), `ld` (local-deleted), `bd` (both-deleted), `cdu` (conflict-deleted-vs-updated).
- Blob: `nb` (no-blob), `cb` (cloud-has-blob), `lb` (local-has-blob), `bb` (both-have-blob), `nh` (neither-has-blob — both sides Staged). Not applicable shown as `—`.
- Watermark (wm): `0` (zero), `p` (partial), `cur` (current), `cR` (cloud-reset), `lR` (local-reset).
- Rounds: `0r`, `1r`, `2r`, `uc` (until-converged).
- Fail / recov: `ok` = no-failure; otherwise listed.

A fresh session picking this up should:

2. Score each candidate ID on three axes (1–3 each), per the strategy laid out before Phase 2 began:
   - **Invariant criticality** — how bad is failure? (data loss / divergence = 3; idempotency on a no-op = 1)
   - **Regression likelihood** — how subtle is the code path? (HLC tie-break, contiguous-prefix blob blocking, watermark fan-out = 3; single-record clean sync = 1)

## Notes / decisions deferred to Phase 3 or implementation

- **Blob retention on tombstone** — *resolved*: tombstones are soft-deletes only. Records and blobs both remain in storage; sync must behave as if the record is deleted. Hard deletion / GC is out of scope.
- **Out-of-band missing blobs** — *resolved and removed*: the prior `blob-key-present-but-missing` dimension value was conflating two things. The current candidates use `neither-has-blob` (both sides Staged) for the legitimate "metadata propagated, blob hasn't" state. Degenerate "someone deleted the file from storage out of band" cases are explicitly out of scope per `system-design.md`'s residency invariant.
- **Residency assertion target**: S3-015 (and any future Staged↔Resident transition test) should assert via `residencyOf` from `packages/sync-engine/src/residency.ts` rather than re-implementing the predicate inline. Note `residencyOf` is async (`(row, localStorage) => Promise<RecordResidency>`) — must be awaited.
- **Tie-break determinism** (S5-011, S7-007) — *resolved*:
  - **SR (S5-011)**: `packages/protocol-primitives/src/hlc/compare.ts` breaks ties by `nodeId` via lexicographic compare (lines 11–12); `maxHLC` returns `a` when `compareHLC(a, b) >= 0` (line 17). Larger `nodeId` wins.
  - **AW (S7-007)**: applier SQL compares the serialized `updated_at` string with strict `>` (`storage-sqlite/src/app-syncable/apply.ts:89`, `storage-aurora-dsql/src/app-syncable/apply.ts:92`). The serialized form is `<wallTime:12>:<counter:4>:<nodeId>` (`packages/protocol-primitives/src/hlc/serialize.ts:6-8`), so lexicographic compare gives the same order as `compareHLC` and the tie-break direction matches SR: **larger nodeId wins** when `(wallTime, counter)` match.
  - **Bytewise-identical `updated_at` between two sides cannot occur** — the nodeId is part of the string and the two sides have distinct nodeIds. The S7-007 candidate is reframed as "identical `(wallTime, counter)`, different nodeIds." Truly-identical strings would require a configuration error (shared nodeId) and are explicitly out of scope.
  - For the standard `"local"` / `"cloud"` fixture: `"local" > "cloud"` → **local wins ties** for both SR and AW. Assertions should hard-code this direction; if fixture nodeIds change, expectations flip with them.
- **Multi-mixed-nodes coverage** — *resolved*: keep coverage in S3-007 (per-nodeId contiguous-prefix under failure) and S6-008 (watermark fan-out within a paginated batch). No dedicated mini-seed needed; the invariant under test is already "different nodeIds advance independently" and both candidates exercise it. If a regression appears, prefer adding focused cases over a new seed.

# Phase 3 — Ranked

Each candidate is scored on three axes (1–3 each):

- **C** = invariant criticality (data loss / divergence = 3; idempotency on a no-op = 1)
- **R** = regression likelihood (HLC tie-break, contiguous-prefix blob blocking, watermark fan-out, multi-round pagination = 3; single-record clean sync = 1)
- **U** = uniqueness vs. higher-scored candidates (3 = exercises a code path nothing above covers; 1 = essentially a mirror of something already kept)

`Score = C + R + U` (max 9). Sorted descending; ties broken by ID. All 103 active candidates are retained — the ≥100 target leaves almost no slack, and the dedup pairs the Phase 2 doc flagged (`S2-015↔S7-*`, `S3-013↔S6-005`, `S5-004↔S5-005`) all turn out to exercise meaningfully distinct code paths once you trace them (different DTs, different operation grammars, different LWW branches). Mirror pairs are kept but marked U=1.

The ranking is intended to drive implementation order, not exclusion: write the 9s first, the 8s next, etc.
