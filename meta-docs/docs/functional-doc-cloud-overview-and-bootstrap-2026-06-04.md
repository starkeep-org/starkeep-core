# Part 1 — Current state

## Overview

Cloud bootstrap is the one-time procedure that prepares a customer's own AWS account to host a Starkeep deployment. It runs before any app, and before the cloud data server itself, exists. After bootstrap the account holds: a Cognito user pool and identity pool that the human admin will log in through; four IAM roles that form the install-time delegation chain (admin → Manager → install-DDL / install-infra); five permissions-boundary managed policies that cap what any future Starkeep role can do; an S3 bucket for Pulumi stack state; an S3 bucket for compiled Lambda bundles ("artifacts"); and an SSM parameter that will hold the Pulumi state-encryption passphrase. Everything else — the cloud-data-server, Starkeep Drive, third-party apps, the DSQL cluster, the files bucket, Cost-and-Usage Reports — is provisioned in *later* phases by code outside this scope.

Bootstrap is shipped to the customer as a single CloudFormation template generated by `generateBootstrapTemplate` (`aws-bootstrap/src/bootstrap/bootstrap-template.ts`). The customer pastes it (or follows a console URL) into CloudFormation in their own account, sets one parameter (`StackPrefix`, default `starkeep`), and deploys. There is no SaaS control plane, no cross-account assume, and — notably — no IAM user.

Three functional concepts organize the rest of Part 1: **how the customer authenticates** (Cognito + STS, not an IAM user); **the role and boundary inventory** the template creates and the delegation chain those roles enable; and **what the template intentionally does *not* create**, which includes both the data plane and Cost-and-Usage Reports.

## How the customer authenticates: Cognito federated login

Bootstrap does not provision an IAM user for the customer. The human admin signs in to a Cognito User Pool (`UserPool` in the template, configured for `ADMIN_CREATE_USER` only, email-as-username, `ALLOW_USER_PASSWORD_AUTH`). A Cognito Identity Pool then exchanges that signed-in identity for temporary STS credentials in the **admin-app role** (`${StackPrefix}-app-admin-role`) via `sts:AssumeRoleWithWebIdentity`. The admin-app role is the *only* identity the human ever directly holds; everything else is reached by further role assumption under those temporary credentials.

This shape is load-bearing for the security stance documented in `roles-and-permissions.md`: there are no long-lived access keys to lose, the human is not a superuser, and every administrative action eventually carries a *different* assumed identity. The bootstrap template enforces it structurally — the `AdminAppRole`'s trust policy only accepts Cognito identities from the identity pool this same stack created, and the `IdentityPoolRoleAttachment` wires the pool's `authenticated` role slot to that role's ARN.

The Cognito user is created by an admin action later (the stack output `ConsoleLink` points the customer at the Cognito user-creation page in the AWS console). Bootstrap itself creates *no* user records — it only stands up the pools.

## The role and boundary inventory

Bootstrap creates **four IAM roles** and **five permissions-boundary managed policies**. The roles form a delegation chain whose shape is the centerpiece of Starkeep's cloud security model.

### The four roles

1. **Admin-app role** (`${StackPrefix}-app-admin-role`). Federated entry point for the human admin. Its inline policy (`admin-app-policy.ts`) grants Cognito user management on its own user pool, S3 read/write on its own admin prefix (`apps/admin/*`), S3 read on the billing bucket, `dsql:DbConnect` (not `DbConnectAdmin`), CloudWatch log writes scoped to admin Lambda log groups, and crucially `sts:AssumeRole` on `${StackPrefix}-manager-role`. It does not carry per-app data-plane power; it is the launching pad for further role assumption.

2. **Manager role** (`${StackPrefix}-manager-role`). The delegation hub. Trusted only by the admin-app role. Its inline policy (`manager-policy.ts`) lets it mint and revoke per-app roles whose names match `${StackPrefix}-app-*` (provided the role is born with one of three allowed permissions boundaries — see below), put/delete inline policies on those roles, attach and detach temporary policies on the install-DDL and install-infra roles, and assume the install-DDL, install-infra, and per-app roles. Manager holds **zero standing data-plane power**: no S3, no DSQL connect, no Lambda. Its job is to delegate, not to act.

3. **Install-DDL role** (`${StackPrefix}-install-ddl-role`). Trusted only by Manager. Permissions boundary: `${StackPrefix}-install-ddl-permissions-boundary`. Standing inline policies: none. Around each app install or uninstall, Manager attaches a short-lived inline policy granting `dsql:DbConnectAdmin`, the install code runs the per-app DDL as PG admin, and Manager detaches the policy. This is the only identity in the deployment that can ever connect to DSQL as PG admin.

4. **Install-infra role** (`${StackPrefix}-install-infra-role`). Trusted only by Manager. Permissions boundary: `${StackPrefix}-install-infra-permissions-boundary`. Standing inline policies: none. Around each install or uninstall, Manager attaches a short-lived inline policy scoped to the specific app being installed (its Lambda names, its log group, its Pulumi state file, and `iam:PassRole` onto its own per-app role). The install code then runs the per-app Pulumi program under this role to create the app's compute stack, after which Manager detaches the policy.

The cloud-data-server role, the per-app roles, the User-Data-Owner role for Starkeep Drive — **none** of these exist after bootstrap. They are minted later by Manager during their respective installs, each born under one of the boundaries bootstrap created. From bootstrap's perspective, every role created is an *install-time* identity; no runtime data-plane identity exists yet.

### The five permissions boundaries

A permissions boundary in IAM is a ceiling: it caps what an identity can do regardless of what's attached to it. Bootstrap creates them as managed policies so future roles can reference them by ARN. The five are:

- **App permissions boundary** (`${StackPrefix}-app-permissions-boundary`). The ceiling every Manager-minted per-app role will be born under. Scopes S3 to the app's own prefix, allows `dsql:DbConnect` (not admin), denies all IAM mutation. The full statement set is in `permissions-boundary.ts`.

- **Foundational permissions boundary** (`${StackPrefix}-foundational-permissions-boundary`). Wider ceiling reserved for the cloud-data-server role. Permits DSQL cluster admin, S3 bucket admin on the well-known `${StackPrefix}-{files,billing}-*` patterns, Lambda/log-group/API-Gateway-v2 admin scoped to cloud-data-server resources, `iam:PassRole` of its own role onto Lambda, and the CUR verbs (`cur:PutReportDefinition`, `cur:DescribeReportDefinitions`, `cur:DeleteReportDefinition`, plus tagging — `foundational-permissions-boundary.ts:241-247`). Denies IAM mutation.

- **User-Data-Owner permissions boundary** (`${StackPrefix}-user-data-owner-permissions-boundary`). Wider-than-per-app ceiling reserved for the future Starkeep Drive role. Permits `dsql:DbConnect` and S3 read/write/list on the files bucket under the shared-data prefix (`shared/*`) — and nothing else: no Lambda, no API Gateway, no DSQL cluster admin, no IAM mutation. The role itself is not created here; it will be minted at Drive install time under this boundary.

- **Install-DDL permissions boundary** (`${StackPrefix}-install-ddl-permissions-boundary`). Narrow ceiling for the install-DDL role: essentially `dsql:DbConnectAdmin` plus a defense-in-depth IAM deny.

- **Install-Infra permissions boundary** (`${StackPrefix}-install-infra-permissions-boundary`). Narrow ceiling for the install-infra role: Lambda admin, log groups, API Gateway v2 integrations/routes, Pulumi state I/O, `iam:PassRole` onto per-app roles, plus a defense-in-depth IAM deny. Per-install scoping to a specific app is the job of the temp policy on top, not the boundary.

The Manager role's `iam:CreateRole` statement (`manager-policy.ts:14-43`) hardcodes an `ArnLike` condition that allows only three of those five boundaries on roles it mints — app, foundational, or user-data-owner — using a wildcard for the account-id segment of the ARN. The install-DDL and install-infra boundaries are not in that allow-list because those roles already exist (bootstrap created them); Manager only puts inline policies on them and never recreates them.

### The delegation chain in one picture

```
Human admin
  └─ Cognito login → STS exchange → AdminAppRole
       └─ sts:AssumeRole → ManagerRole (no data-plane power)
            ├─ sts:AssumeRole → InstallDdlRole  (PG admin, temp-scoped)
            ├─ sts:AssumeRole → InstallInfraRole (per-app compute, temp-scoped)
            └─ sts:AssumeRole → app-<appId>-role (post-mint verification)
```

Mechanics of how a per-app role is later assumed by the cloud-data-server at request time — and how DSQL maps assumed IAM identities to PG roles — live with the `cloud-server-auth` and `cloud-server-install` topics, not here.

## Supporting resources bootstrap creates

Three resources that are not roles or boundaries also live in the bootstrap template:

- **Pulumi state bucket** (`${StackPrefix}-pulumi-state-${AWS::AccountId}-${AWS::Region}`, versioned). Holds per-app Pulumi stack state. Read and written by install-infra under its temp policy.
- **Artifacts bucket** (`${StackPrefix}-artifacts-${AWS::AccountId}-${AWS::Region}`, versioned). Holds each app's compiled Lambda bundle. Read by `aws.lambda.Function` at function create/update, written by install-infra during install, deleted by install-infra on uninstall. Never holds user data.
- **Pulumi passphrase SSM parameter** (`/${StackPrefix}/pulumi/passphrase`). The encryption passphrase for Pulumi state. Bootstrap writes the literal string `REPLACE_WITH_RANDOM_32_BYTE_VALUE` and the resource description tells the operator to replace it manually or by a post-deploy custom resource (see Part 2: Missing behaviors).

## Stack outputs

The template emits the IDs/ARNs the next phase needs: `UserPoolId`, `UserPoolClientId`, `IdentityPoolId`, `AdminAppRoleArn`, `ManagerRoleArn`, the three boundary ARNs (`AppPermissionsBoundaryArn`, `AppFoundationalPermissionsBoundaryArn`, `UserDataOwnerPermissionsBoundaryArn`), `InstallDdlRoleArn`, `InstallInfraRoleArn`, the two bucket names, the `Region` the stack was created in, the `StackPrefix` actually used, and a `ConsoleLink` that points to the Cognito user-creation page so the admin can create their initial login. There is no output for the install-DDL or install-infra *boundary* ARNs — they are referenced internally and never need to be quoted by other components.

Two helper functions in the same file shape how the template is delivered:

- `getCloudFormationCreateStackUrl(region, opts?)` returns a deep link into the CloudFormation console's create-stack page. The customer pastes (or uploads) the template body and runs the stack themselves. When `opts.stackName` is supplied, it is appended as a `stackName` fragment query param so the console pre-fills the "Stack name" field — admin-web passes `` `${stackPrefix}-bootstrap` `` here so the operator does not have to type the name (and the convention `getBootstrapStackOutputsUrl` later searches for is enforced rather than instructed).
- `getBootstrapStackOutputsUrl(region, stackName="starkeep-bootstrap")` returns a deep link into the CloudFormation stacks list filtered to the bootstrap stack (substring `filteringText` match on the name) so the customer (or the admin UI) can read its outputs after deployment.

## Cost and Usage Reports at the bootstrap layer

The CUR child topic is partly addressed by bootstrap and partly deferred. What bootstrap does for CUR is purely capability reservation: the foundational permissions boundary includes the `cur:*` verbs the cloud-data-server role will need (`foundational-permissions-boundary.ts:241-247`), so when that role is later minted under that boundary it can create and manage report definitions. Bootstrap does **not** create the billing bucket, the CUR report definition, the textORcsv/GZIP/OVERWRITE_REPORT configuration, or the us-east-1 provider that CUR requires — those live in the cloud-data-server's Pulumi program (`starkeep-core/packages/admin-installer/src/builtin-programs/cloud-data-server-program.ts`), which is in the `cloud-server-install` topic, not this one.

For an operator's mental model: at the end of bootstrap, no Cost-and-Usage Reports have been requested from AWS yet — only the *permission* to request them later has been engraved into a managed policy ARN.

## What bootstrap intentionally does not create

A new reader is most likely to be surprised by what's *missing* from the post-bootstrap state. None of these are bugs; each is a deliberate seam between phases.

- No data plane: no DSQL cluster, no files bucket, no billing bucket, no API Gateway, no Lambdas.
- No runtime app identities: no cloud-data-server role, no per-app roles, no Starkeep Drive (User-Data-Owner) role. Drive's *boundary* exists; its *role* is minted at Drive install.
- No Cost-and-Usage Reports (see previous section).
- No CodeBuild project, no SaaS control-plane resources, no cross-account assume roles — Starkeep is single-account-only and `roles-and-permissions.md` makes the customer's own AWS account the trust boundary.
