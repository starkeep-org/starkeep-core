# Plan: CloudFront for the cloud deployment (app assets + shared-data file bytes)

Date: 2026-07-10
Status: implemented 2026-07-11 (Parts A + B); pre-implementation investigation completed 2026-07-10.
Verification steps (§ "Part A steps" 3, "Part B steps" 5) are live-deploy checks still
to be run against a real cloud install; all code, IAM/trust decisions, and unit/adversarial
tests are in place and green.
Scope: starkeep-core cloud deployment (admin-installer / cloud-data-server), with knock-on
changes in cloud apps' client URL plumbing.

## Goal

Put a CloudFront distribution in front of the cloud deployment so that:

1. **Part A — app static assets** (the Next.js JS/CSS/HTML each cloud app serves under
   `/apps/{appId}/*`) are edge-cached instead of being generated by a Lambda invocation
   per request.
2. **Part B — shared-data file bytes** (photo originals/thumbnails and any other
   file-backed shared records) are served through CloudFront signed URLs with edge
   caching, instead of uncached S3 presigned GETs.

The two parts share one distribution but are otherwise independent designs with
independent risk profiles. **Do Part A first, then Part B as its own step.** Part A is
config-heavy but introduces no new secrets or trust-boundary changes; Part B's only
genuinely new machinery is the CloudFront key-pair lifecycle, and it carries a real
(analyzed and accepted, see below) trust-boundary change.

## Current state (verified 2026-07-10)

- Cloud apps serve their SPA via a manifest-declared `static` compute handler
  (`ANY /{proxy+}`, `auth: "public"`) on the shared API Gateway, mounted under
  `/apps/{appId}` (see `starkeep-apps/photos/starkeep.manifest.json`).
- All `/apps/{appId}/*` API routes are HMAC-signed (signed server-side by
  `@starkeep/app-client` in per-app Lambdas); browser-originated calls are JWT-gated
  (`packages/admin-installer/builtin-apps/cloud-data-server/src/api-handler.ts`).
- File bytes: `GET /data/records/:id/file-url` and the batch
  `POST /data/records/file-urls` mint S3 presigned GET URLs, `expiresIn: 3600`
  (clamped to remaining STS session TTL), signed per-request by the cloud-data-server
  Lambda **under the assumed per-app role's credentials** (api-handler.ts ~518, ~727;
  `packages/storage-s3/src/adapter.ts:190`).
- **Object key layout (confirmed):** shared blobs live at
  `shared/<category>/<shard>/<hash>` (api-handler.ts:518). Keys are content-addressed →
  objects are immutable. App-specific files live under `apps/<appId>/syncable/...` and
  are served by the separate `/app-data/files` presign flow.
- **SSE (investigated — non-issue):** the photos app's EventSource freshness path is
  local-only. The strategy is decided at boot from runtime config: cloud builds
  (`apiGatewayUrl` set) use 30-second polling, never SSE
  (`starkeep-apps/photos/src/lib/usePhotoFreshness.ts:49`). No streaming response ever
  crosses the gateway, so CloudFront's response-buffering/timeout behavior is not a
  constraint for Part A.
- The manifest schema already defines a `cloudfront` shared-resource kind
  (`packages/admin-manifest/src/schema.ts:25`) but **nothing implements it** — it is
  schema-only. This plan deliberately does NOT implement the generic
  `sharedResources: cloudfront` machinery; the distribution is platform-owned.

### Gateway-URL plumbing inventory (traced 2026-07-10)

Source of truth and fan-out for the public base URL:

1. **Origin:** the cloud-data-server Pulumi program exports `apiGatewayUrl` as a stack
   output (`builtin-programs/cloud-data-server-program.ts:494`).
2. **Persisted:** `builtin-installs.ts:373` captures it into the installer config file
   (`~/.starkeep/config.json`, via `app-cli-config.ts` / `configPath`).
3. **Browser-facing fan-out:**
   - `cli-install-app.ts:199` fills manifest handler-env placeholders
     (`STARKEEP_API_GATEWAY_URL`, plus user-pool ids) for each app's Lambdas.
   - `packages/app-client/src/runtime-config.ts:18` serves those env values to the
     browser as `/starkeep-runtime-config.json` (fetched by the SPA:
     `starkeep-apps/photos/src/lib/runtime-config.ts:21`).
   - The SPA builds API URLs as `${apiGatewayUrl}/apps/photos`
     (`starkeep-apps/photos/src/lib/data-client.ts:117`); presence of `apiGatewayUrl`
     is also the cloud/local mode switch (data-client.ts:26,
     usePhotoFreshness.ts:49).
   - admin-web reads the same config value (`app/api/exec/deploy-outputs/route.ts:21`,
     `src/lib/cloud-config.ts`) for its cloud-mode data source.
4. **Server-to-server fan-out (stays on the raw gateway URL — see decision below):**
   - `pulumi-program.ts:61` sets `STARKEEP_CLOUD_DATA_BASE = ctx.apiGatewayUrl` in every
     per-app Lambda for HMAC-signed app-client calls.
   - local-data-server's cloud broker/sync paths read `config.apiGatewayUrl`
     (`apps/local-data-server/server.ts`).

## Part A — app static assets

### Design (all decisions settled)

One platform-owned CloudFront distribution, created in the core cloud deployment's
Pulumi program (alongside the cloud-data-server resources, so `pulumi destroy` tears it
down — no `teardown-bootstrap.sh` change needed for Part A).

- **Origin:** the API Gateway execute-api stage domain.
- **Domain:** the default `*.cloudfront.net` domain. No custom domain, no ACM cert
  (avoids the us-east-1 cert requirement entirely for now).
- **Cache behaviors and policies (settled):**
  - `/apps/*/_next/static/*` → managed `CachingOptimized` cache policy, no origin
    request policy needed. These filenames are content-hashed by Next.js (immutable),
    safe for long TTLs, no invalidation story needed. These are public assets — no auth
    headers involved.
  - Default behavior → managed `CachingDisabled` cache policy + managed
    **`AllViewerExceptHostHeader`** origin request policy. This combination resolves
    the two known CloudFront gotchas at design time: (a) with caching disabled,
    `Authorization` and custom headers (the HMAC signature headers) are only forwarded
    if the origin request policy includes them — `AllViewerExceptHostHeader` forwards
    all viewer headers; (b) forwarding the viewer `Host` header breaks API Gateway
    HTTP APIs (the gateway requires its own execute-api hostname), which is exactly why
    the `...ExceptHostHeader` variant exists. HMAC signatures sign over the path, which
    CloudFront preserves, so signatures remain valid.
- **HTML entry points** (`/apps/photos/`, `404.html`) are not content-hashed → they fall
  under the default (no-cache) behavior, which is correct as-is; no CreateInvalidation
  step needed anywhere.

### URL plumbing decision (settled)

**Browser-facing traffic moves to the CloudFront domain; server-to-server traffic stays
on the raw gateway URL.**

- Add a `publicBaseUrl` stack output (the distribution domain) next to `apiGatewayUrl`
  and persist both in `~/.starkeep/config.json`.
- `cli-install-app.ts`'s `platformEnv` fills `STARKEEP_API_GATEWAY_URL` with
  `publicBaseUrl` — this single change repoints the entire browser fan-out (runtime
  config → SPA data-client / cloud-config / admin-web), since everything downstream
  derives from that env value. The SPA's same-origin `withBasePath` invariant is
  unaffected (single origin is preserved; it just becomes the CloudFront domain).
- `STARKEEP_CLOUD_DATA_BASE` (Lambda→gateway HMAC calls) and the local-data-server
  broker keep using `apiGatewayUrl` directly: machine-to-machine calls gain nothing
  from the edge and this keeps the sync/broker path independent of CloudFront
  availability and timeout characteristics.

### Trust posture (checked against data-roles-and-permissions.md, 2026-07-10)

Part A changes **no** trust boundary. The roles doc's request-path description
("The client calls the shared API Gateway with a Cognito JWT and an app identification
on the request", data-roles-and-permissions.md:118) still holds — the JWT authorizer
and HMAC verification remain on the gateway/handler exactly as before; CloudFront is a
pass-through in front. Two documentation-level notes to make when implementing:

- The gateway URL remains directly reachable (locking an HTTP API to CloudFront-only
  requires extra machinery we are not adding). CloudFront is therefore an
  **optimization layer, not a security boundary** — state this explicitly in the roles
  doc so nobody later assumes edge-level protection.
- Update the entry-point wording at data-roles-and-permissions.md:118 to mention that
  browsers reach the gateway via the distribution.

### Part A steps

1. Add the distribution to the core Pulumi program: gateway origin, the two behaviors
   with the managed policies named above. Export `publicBaseUrl`.
2. Persist `publicBaseUrl` in the installer config; point `platformEnv`'s
   `STARKEEP_API_GATEWAY_URL` at it (`cli-install-app.ts:199`).
3. Verify end to end (cloud photos app): SPA loads via CloudFront; `_next/static/*`
   returns `x-cache: Hit from cloudfront` on second fetch; JWT-gated browser calls and
   HMAC-signed Lambda→gateway calls succeed; 30s polling freshness works.
4. Make the two roles-doc wording updates above.

### Remaining Part A risks (verification-time, not design-time)

- Deploy latency: distribution create/update takes minutes; the install flow already
  tolerates long Pulumi operations.
- The managed-policy combination is settled on documented CloudFront semantics; the
  e2e test in step 3 is the confirmation, not an open design question.

## Part B — shared-data file bytes via CloudFront signed URLs

### Design (all decisions settled)

- **Second origin on the same distribution:** the shared-data S3 files bucket, locked
  with Origin Access Control (OAC); bucket policy admits only the distribution.
- **Behavior:** path pattern **`shared/*`** (matches the confirmed key layout
  `shared/<category>/<shard>/<hash>`; no collision with gateway routes, which live
  under `/apps/*`). Long TTL — objects are content-addressed and immutable; thumbnails
  especially are re-requested across sessions and devices. **Scope deliberately
  excludes `apps/*`** (app-specific files) — see trust analysis.
- **Cache-key discipline:** custom cache policy for this behavior that excludes query
  strings from the cache key (the signed-URL params `Expires`/`Signature`/`Key-Pair-Id`
  are validated by CloudFront, then caching is by path). Without this every freshly
  signed URL is a cache miss and Part B delivers nothing.
- **Access control — CloudFront signed URLs replace S3 presigned URLs** on the shared
  file read path only:
  - RSA key pair; public key registered as a CloudFront public key + key group; the
    `shared/*` behavior requires signed requests from that key group.
  - Private key stored as an SSM SecureString; the cloud-data-server Lambda reads it
    once per warm container and signs with `@aws-sdk/cloudfront-signer`.
  - The shared file-url endpoints (`GET /data/records/:id/file-url`, batch
    `POST /data/records/file-urls`, and the shared-prefix GET
    `/files/{key}/presign` route) keep their existing authorization checks (app
    grants, type readability) and response shapes — only the URL-minting call changes.
  - **Unchanged:** presigned PUT uploads (`POST /files/presign`) — CloudFront is
    read-path only; and the `/app-data/files` flow for `apps/<appId>/syncable/*`
    private files, which stays on S3 presign entirely.
- **TTL decision (settled):** signed URLs get a flat 3600s expiry. The existing
  STS-remaining-TTL clamp is dropped for the CloudFront-signed path — it exists because
  S3 presigned URLs die with the signing session's STS credentials, which does not
  apply to CloudFront key-pair signing. (The clamp stays on the untouched presign
  paths.)

### Trust-boundary analysis (done 2026-07-10 against data-roles-and-permissions.md)

The roles doc's core principle is **layered nested ceilings** (line 40): IAM (category-
granular), PG GRANTs, and the application-layer type-granular check each bound the
others' failures. Today's read path inherits that: the handler presigns S3 URLs *under
the assumed per-app role's credentials*, so even if the application-layer grant check
had a bug, the per-app permissions boundary (category-scoped S3 prefixes,
data-roles-and-permissions.md:64,118-120) caps what a presigned URL can reference.

**Part B removes the IAM layer from the shared-read path.** The CloudFront signing key
is not category-scoped: whatever path the application code signs, CloudFront serves.
A bug in the application-layer check would no longer be bounded by the per-app role's
S3 ceiling. This is a real regression of the layering principle, mitigated and accepted
as follows:

**Threat model, including malicious apps (clarified 2026-07-11):** "application-layer
bug" here means a bug in the *platform's* authorization code (cloud-data-server), and
the adversary it matters against includes a malicious app — one that declares narrow
manifest grants but tries to read all shared data. The signing private key never
touches app code (SSM read is granted only to the cloud-data-server Lambda role), so a
malicious app cannot sign URLs itself; it can only ask the platform's file-url
endpoints, which check the caller's install-time grants (per-type on the record-id
routes; per-category on the raw-key route via `parseObjectKey`, since raw keys only
encode the category). Malice alone therefore exfiltrates nothing beyond the app's
declared grants — which is the exposure accepted at install time under any signing
mechanism. What Part B changes is the *payoff of a platform authz bug* for such an
app: today an exploited bug is IAM-capped at the caller's granted categories; after
Part B it reaches anything under `shared/*`. The app-layer check becomes the sole
per-request line of defense for shared reads, which motivates the pre-sign
revalidation chokepoint below.

- **Blast radius is capped structurally at `shared/*`** by the behavior's path pattern:
  the distribution cannot serve `apps/*` (private app data) or any other prefix no
  matter what gets signed, because no behavior routes there and OAC only opens the
  bucket to the distribution. The worst case of an application-layer bug is therefore a
  cross-type leak *within shared data* — data every installed app could already be
  granted by manifest — not exposure of private app files or other buckets.
- The write path is untouched and remains fully IAM-bounded (per-app presigned PUTs
  under category-scoped prefixes).
- The application-layer grant check is the same code that already makes the
  type-granular cut the coarser layers can't (roles doc line 125); it is exercised by
  every read today.
- **Pre-sign revalidation chokepoint (decided 2026-07-11):** immediately before every
  CloudFront signing call, one dedicated function re-validates the object key against
  the caller's grants — parse the key, require the `shared/` namespace, and check the
  category segment against the caller's granted categories. This is in-process
  (grants are already in memory for the request's existing auth check), so it adds no
  measurable latency — microseconds, versus ~1ms for the RSA signature itself. It is
  the same trust domain as the route-level checks, so it is weaker than the IAM layer
  it replaces, but it catches route-level bugs (a route forgetting or mis-wiring its
  check) at a single auditable spot, and it is the only per-request defense line left
  on this path.
- Accepted for the current deployment model (single-admin personal deployment). If the
  layering ever needs restoring, the escape hatch is per-category key groups or
  reverting hot categories to S3 presign — both compose with this design without
  rework.
- **Per-category key groups considered and rejected for now (2026-07-11):** they would
  restore a category-granular ceiling independent of application code, but the cost is
  ongoing, not one-time — N key pairs/SSM params/rotations instead of one; a
  key-selection step in the signer that is itself the class of code being defended
  against; and every new category becomes a core Pulumi change plus a minutes-long
  CloudFront distribution update (with a default quota of 4 key groups per
  distribution). The single-key design absorbs new categories with zero infrastructure
  change: the `shared/*` behavior covers them structurally and the grant check +
  chokepoint pick them up at runtime, preserving the property that data-type additions
  are data-server/registry/manifest changes only. Key groups remain the documented
  escape hatch above and compose later without rework.

**Required doc update (part of Part B's implementation, content decided now):** add a
paragraph to data-roles-and-permissions.md's read-path section stating that shared file
bytes are served via CloudFront signed URLs; that the signing ceiling is the `shared/*`
behavior scope (structural) plus the application-layer grant check (type-granular);
and that the per-app IAM ceiling no longer applies to shared *reads* (it still applies
to writes and to app-data files).

### Key-pair lifecycle (settled)

- **Creation:** generated inside the core Pulumi program (key material via Pulumi's
  TLS/crypto support), public key + key group registered on the distribution, private
  key written to an SSM SecureString parameter; the cloud-data-server Lambda role gets
  `ssm:GetParameter` on it. Everything rides the stack → teardown is `pulumi destroy`,
  **no `teardown-bootstrap.sh` change needed**. (If during implementation any piece is
  forced outside the stack, the teardown script must be updated in lockstep — standing
  convention — but the design keeps everything in-stack precisely to avoid that.)
- **Rotation:** out of scope (fresh-start philosophy), but the key group is created
  able to hold two keys so rotation needs no redesign.

### Part B steps

1. Key pair in Pulumi: generate, register public key + key group, private key → SSM
   SecureString, grant the cloud-data-server Lambda role read on it.
2. Add the S3 origin + OAC + `shared/*` behavior (long TTL, key-group-signed requests
   required, query strings excluded from cache key); update the bucket policy.
3. Swap URL minting in the shared file-url routes from S3 presign to CloudFront
   signing (auth checks unchanged; drop the STS clamp on this path only). All signing
   goes through the pre-sign revalidation chokepoint from the trust analysis: one
   function that re-checks the key's namespace and category against the caller's
   grants right before signing — no route may reach the signer around it.
4. **Harden the now-load-bearing authorization code (added 2026-07-11):** with the IAM
   backstop gone, `parseObjectKey` and the grant checks on every shared file-url route
   (record-id routes, batch route, raw-key route) carry the whole per-request security
   weight for shared reads. Before enabling the CloudFront path:
   - Do a focused security review of `parseObjectKey` and each route's check wiring —
     note the raw-key route is only category-granular (keys don't encode full type
     ids), so confirm that granularity is acceptable per route and that no route
     reaches the signer without a check.
   - Add automated tests treating this as adversarial input handling, not just happy
     path: foreign categories, `apps/*` and unknown-namespace keys, path traversal
     (`..`, embedded `/`), URL-encoding and double-encoding tricks, empty/short
     segments, case variants, and the chokepoint being unbypassable from every
     signing route.
5. Verify: photo bytes load in the cloud photos app; second fetch of the same object is
   an edge hit; a tampered signature is rejected; an app without a grant for the type
   still cannot obtain a URL; the adversarial tests from step 4 pass;
   `apps/*` keys are unreachable through the distribution;
   uploads (presigned PUT) and app-data file reads unaffected.
6. Apply the data-roles-and-permissions.md update specified in the trust analysis.

## Explicit non-goals (this plan)

- Implementing the generic `sharedResources: cloudfront` manifest machinery.
- Moving static assets to an S3 origin (upload-at-install pipeline). Part A already
  captures most of the latency win via edge caching; revisit if Lambda invocation cost
  for assets becomes material.
- Serving `apps/<appId>/syncable/*` app-data files via CloudFront (stays on S3
  presign — deliberate trust-scoping decision, not an oversight).
- Locking the API Gateway to CloudFront-only access.
- Custom domain / ACM certificate.
- Key rotation automation.
- WAF, logging, price-class tuning (defaults fine for a personal deployment).
